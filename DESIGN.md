# Task Graph 插件设计

> 状态：设计稿 v0.1  
> 日期：2026-06-28  
> 目标：把 Obsidian Tasks 任务转换为清晰、可操作的项目依赖图。

## 1. 产品定位

Task Graph 不是另一个“全库任务列表”，也不是 Obsidian 原生关系图的换皮。

它是一张以项目为单位的执行地图，主要回答三个问题：

1. 这个项目由哪些任务组成？
2. 哪些任务被前置任务阻塞？
3. 我现在可以立即开始哪一项？

推荐名称为 **Task Graph**，插件 ID 使用 `task-graph`。它与 Time Grid 保持视觉一致，但作为独立插件开发、安装和维护。

## 2. 核心设计原则

- **项目优先**：默认一次只看一个文件、文件夹或保存的项目视图，不默认铺开整个 vault。
- **任务文本是唯一事实来源**：图中的状态和依赖关系都来自 Markdown，不另建一套任务数据库。
- **先读后写**：MVP 先保证解析、查看、筛选和定位可靠，再加入改依赖等高风险写操作。
- **关系有明确方向**：前置任务指向被阻塞任务，阅读方向默认从左到右。
- **信息克制**：颜色只表达状态和阻塞，不按每个标签随机染色。
- **无 Dataview 强依赖**：直接读取 Markdown，Tasks 插件存在时增强编辑能力，不存在时仍可查看。

## 3. 面向当前任务格式的数据约定

当前任务格式已经包含构图所需的核心信息：

```md
- [ ] TimeGrid 🆔 9v4fal
- [ ] 找到所有不够动态的部分 🆔 0lxlu3 ⛔ 9v4fal
- [ ] 日历文字响应式调整 🆔 339azh ⛔ 9v4fal
```

解析规则：

| 内容 | 图中含义 |
| --- | --- |
| `🆔 9v4fal` | 节点稳定 ID |
| `⛔ 9v4fal` | 当前任务被 `9v4fal` 阻塞 |
| `- [ ]` | 待办 |
| `- [/]` | 进行中 |
| `- [x]` | 已完成 |
| `- [-]` | 已取消 |
| Tasks 日期与优先级 emoji | 节点元数据 |
| `#tag` | 筛选标签 |

边的方向统一为：

```text
前置任务 ──────> 依赖它的任务
```

没有 `🆔` 的任务仍可显示，但只使用“文件路径 + 行内容摘要”作为临时身份，不能建立可靠依赖。节点菜单提供“补充任务 ID”入口。

缩进子任务可作为第二种关系：

- `⛔` 关系：实线箭头，表示真正的执行依赖。
- Markdown 缩进：虚线，表示结构归属，不自动等同于阻塞。

## 4. 推荐界面

### 4.1 主区域

插件打开在 Obsidian 主编辑区，而不是右侧窄栏。整体由三层组成：

```text
┌──────────────────────────────────────────────────────────────┐
│ Task Graph  [项目：TimeGrid ▾]  [搜索]  待办  进行中  已完成 │
│             [聚焦分支] [显示完成] [自动布局] [适应画布]       │
├──────────────────────────────────────────────┬───────────────┤
│                                              │ 任务详情      │
│    ○ 项目根节点 ─────> ◉ 可执行任务           │ 状态 / 日期   │
│          │                    │               │ 来源 / 标签   │
│          └──────> ◌ 被阻塞任务 ─────> ✓ 完成  │ 前置 / 后续   │
│                                              │ [打开原文]    │
├──────────────────────────────────────────────┴───────────────┤
│ 12 个任务 · 4 已完成 · 2 进行中 · 3 可开始 · 1 个关系异常     │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 节点

节点建议固定宽度约 220px，高度随内容小幅变化：

```text
┌─────────────────────────┐
│ ◉ 进行中          高优先 │
│ 日历文字响应式调整       │
│ TimeGrid.md · 今天       │
│ #UI  #plugin             │
└─────────────────────────┘
```

节点状态：

- **可开始**：待办且所有前置任务均完成；使用主题强调色描边。
- **进行中**：使用更明显的强调色和轻微外发光。
- **被阻塞**：降低饱和度，并显示锁形图标与未完成前置数量。
- **已完成**：降低透明度，文本可加删除线，但仍保留上下文。
- **逾期**：只在日期文字和一个小警示点上使用危险色，不把整张卡染红。
- **关系异常**：缺失 ID、引用不存在或循环依赖时使用警告色。

### 4.3 详情面板

单击节点后从右侧打开详情面板，显示：

- 完整任务文本
- 当前状态、优先级、开始日期、截止日期
- 所在文件与行附近的标题路径
- 前置任务和后续任务
- “打开原文”“切换完成状态”“聚焦此分支”

MVP 不在面板中提供自由文本编辑，避免把 Markdown 的完整语法悄悄改坏。

## 5. 核心交互

### 必须有

1. 从左侧 ribbon 或命令面板打开 Task Graph。
2. 选择文件、文件夹或已保存的项目视图。
3. 画布支持平移、缩放、适应窗口和自动布局。
4. 单击节点查看详情；双击节点打开原始 Markdown 位置。
5. 直接切换任务完成/未完成状态。
6. 搜索任务，并选择是否同时显示其上游或下游关系。
7. 按状态、标签、文件筛选。
8. 一键聚焦某个节点的完整分支。
9. 显示“可开始”“被阻塞”“关系异常”的数量。

### 第二阶段再做

- 拖动连接点创建或删除 `⛔` 依赖。
- 在图中创建子任务。
- 批量修改状态和标签。
- 将当前筛选保存为 Markdown 中的 `task-graph` 嵌入块。
- 关键路径、预计耗时和项目完成时间推算。

## 6. 与 Tasks Map 的取舍

借鉴 Tasks Map：

- 有向节点图、自动布局、平移缩放。
- 搜索结果可带上游/下游任务。
- 按文件、状态、标签筛选。
- 节点可回到原文，图上可切换完成状态。

不照搬的部分：

- 不把全库未关联任务全部塞入一个侧栏。
- 不让随机标签颜色成为主要视觉编码。
- 不在第一版同时支持 inline task、note task、嵌入图和复杂边编辑。
- 不依赖 Dataview 才能启动。
- 不把“能连线”作为核心价值，把“当前可执行任务”放在第一位。

Task Graph 的差异化重点是：

> **项目聚焦 + 阻塞判断 + 下一步行动，而不是单纯把任务画成节点。**

## 7. 布局算法

默认采用从左到右的分层 DAG 布局：

- 根任务在最左侧。
- 前置任务永远位于依赖任务左侧。
- 同层节点按任务在原文件中的顺序排列，减少每次刷新时的位置跳动。
- 多个互不相连的任务放入底部“未关联”泳道，可折叠。
- 检测到循环依赖时，不继续强行计算层级；将循环节点组合成警告区域。

布局模式：

1. **依赖图**：默认模式，适合项目执行。
2. **树状图**：只显示主父级关系，适合查看任务拆分。
3. **自由布局**：第二阶段支持，保存用户拖动位置。

技术上推荐 `@dagrejs/dagre` 只负责坐标计算，节点使用原生 HTML 卡片，边使用 SVG。这样可以保持较轻的包体，同时比纯 SVG 更容易实现文本、按钮、主题适配与无障碍交互。

### 8.3 索引与更新

- 首次打开时扫描所选范围，而不是插件启动时无条件解析整个 vault。
- 监听 Obsidian vault 的创建、修改、删除和重命名事件。
- 文件修改后只重新解析该文件，并对短时间连续变更做 debounce。
- 写回状态时使用 `vault.process()`，重新按任务 ID 定位最新行，避免旧行号覆盖新内容。
- 图刷新时保留视口、选中节点和用户手动移动过的节点。

## 9. 核心模型

```ts
type TaskStatus = 'todo' | 'in_progress' | 'done' | 'canceled';

interface TaskNode {
  id: string;
  stableId: boolean;
  text: string;
  status: TaskStatus;
  path: string;
  line: number;
  headingPath: string[];
  tags: string[];
  blockedByIds: string[];
  priority?: 'highest' | 'high' | 'medium' | 'low' | 'lowest';
  startDate?: string;
  dueDate?: string;
}

interface DerivedTaskState {
  readiness: 'ready' | 'blocked' | 'active' | 'finished';
  unresolvedDependencyIds: string[];
  dependentIds: string[];
  warning?: 'missing-id' | 'missing-reference' | 'cycle';
}
```

“可开始”的判断：

```text
任务未完成
AND 任务不是进行中
AND 所有 blockedBy 指向的任务都已完成
```

没有前置任务的待办默认视为可开始。统计项目进度时按节点 ID 去重，避免一个任务被多个节点引用后重复计数。

## 10. 设置项

第一版只保留必要设置：

| 设置 | 默认值 |
| --- | --- |
| 默认项目范围 | 最近一次使用 |
| 布局方向 | 从左到右 |
| 默认显示已完成任务 | 开启，但降低透明度 |
| 未关联任务区域 | 折叠 |
| 显示标签 | 开启 |
| 双击节点行为 | 打开原文 |
| 自动刷新延迟 | 300ms |

项目筛选预设单独保存在插件 `data.json` 中，包含名称、文件/文件夹范围、状态过滤和画布视口，不修改用户笔记。

## 11. 视觉规范

延续 Time Grid 的视觉语言：

- 背景、边框、文字和强调色全部使用 Obsidian CSS 变量。
- 8px 面板圆角、1px 边框、克制阴影。
- 字号以 12px、14px、16px 为主。
- 使用 `var(--interactive-accent)` 表示当前行动。
- 卡片背景使用 `var(--background-primary)`，画布使用 `var(--background-secondary)`。
- 深色和浅色主题都不写死整套颜色。
- 动效只用于选中、聚焦和重新布局，持续 120–200ms。

## 12. MVP 边界

第一版包含：

- 原生解析 Obsidian Tasks inline task。
- 识别 `🆔`、`⛔`、状态、日期、优先级和标签。
- 文件/文件夹范围选择与保存预设。
- 自动依赖图、状态筛选、搜索、分支聚焦。
- 可开始/被阻塞计算。
- 打开原文和切换完成状态。
- 增量刷新、循环与无效引用提示。

第一版明确不包含：

- Note-as-task。
- 图内新增/删除任务。
- 图内拖线修改依赖。
- Markdown 嵌入图。
- AI 自动拆分任务。
- 与 Time Grid 合并为同一个插件。

## 13. 实施顺序与验收

### Phase 1：可靠地“看”

- 建立插件骨架、解析器、索引器和图模型。
- 用现有含 `🆔` / `⛔` 的项目文件验证关系。
- 完成 DAG 布局、节点状态、平移缩放和打开原文。

验收：项目任务数量、状态和每条依赖边都与 Markdown 一致；修改文件后图能自动更新。

### Phase 2：安全地“操作”

- 切换完成状态。
- 文件、状态和标签筛选。
- 搜索、上下游展开、项目预设和详情面板。

验收：图中切换状态后只修改目标任务，原有 Tasks 元数据保持不变。

### Phase 3：真正提升执行效率

- 可开始任务突出显示。
- 阻塞原因、项目进度、关系异常和循环检测。
- 节点位置记忆与更稳定的增量布局。

验收：打开项目图后，不进入原文也能判断下一项行动和主要阻塞点。

## 14. 最终建议

先做一个窄而完整的 Task Graph：

- 只服务 inline Tasks；
- 只把 `🆔` / `⛔` 当作正式依赖；
- 默认按项目文件查看；
- 把“可开始任务”做成最醒目的信息；
- 第一版不做拖线改笔记。

这条路线比直接复制 Tasks Map 更贴合当前使用方式，也能显著降低第一版的解析、写回和交互复杂度。
