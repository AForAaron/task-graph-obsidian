# Task Graph 插件设计

> 状态：当前实现基线 v0.4.0<br>
> 更新日期：2026-07-01<br>
> 目标：把 Markdown Tasks 转换为清晰、可操作、可安全写回的项目依赖图。

## 1. 产品定位

Task Graph 不是全库任务列表，也不是 Obsidian 原生关系图的换皮。它是一张以用户选择来源为边界的
项目执行地图，主要回答：

1. 这张地图包含哪些任务？
2. 哪些任务被前置任务阻塞？
3. 哪些任务现在可以开始？
4. 如何直接在图中调整任务及其关系？

插件 ID 为 `task-graph`。它与 Time Grid 可以共享视觉语言，但作为独立插件开发、安装和维护。

## 2. 设计原则

- **来源明确**：每张地图只读取用户选择的文件和递归文件夹。
- **Markdown 优先**：任务文本、状态与依赖都以笔记原文为准。
- **地图配置独立**：节点位置、视口和来源保存在插件数据中，不污染 Markdown。
- **关系方向稳定**：前置任务始终指向依赖任务，默认从左到右阅读。
- **写回可解释**：删除任务、解除关系等操作必须明确提示实际修改内容。
- **渐进增强**：没有 Tasks 插件时仍可解析和查看；Tasks API 可用时启用完整表单和状态操作。
- **异常可见**：缺失 ID、重复 ID、缺失引用和循环依赖不能静默忽略。

## 3. Markdown 数据约定

```md
- [ ] 项目根任务 🆔 root001
- [/] 设计界面 🆔 design01 ⛔ root001 🛫 2026-06-30
- [ ] 开发画布 🆔 canvas01 ⛔ root001,design01 📅 2026-07-05
```

| 语法 | 含义 |
| --- | --- |
| `🆔 root001` | 稳定任务 ID |
| `⛔ root001,design01` | 当前任务依赖的前置任务 |
| `- [ ]` | 待办 |
| `- [/]` | 进行中 |
| `- [x]` | 已完成 |
| `- [-]` | 已取消 |
| `➕ 🛫 ⏳ 📅 ✅` | 创建、开始、计划、截止与完成日期 |
| `⏫ 🔼 🔺 🔽 ⏬` | 优先级 |
| `#tag` | 标签 |
| `⭐` | 星标 |
| `📄 [[path|任务内容]]` | 关联的 Markdown 内容文件 |

正式边方向：

```text
前置任务 ──────> 依赖它的任务
```

独立任务可以没有 ID。任务一旦参与依赖，就需要唯一 ID。无 ID 节点使用“路径 + 任务正文指纹”
保存临时布局；重复 ID 不会被当作可靠关系目标。

## 4. 地图模型

每张地图包含：

- 唯一 ID 和用户名称；
- 多个文件或递归文件夹来源；
- 以稳定任务键索引的节点位置；
- 平移、缩放后的画布视口。
- 内容文件路径及其展开状态。

全局插件数据还保存当前地图、是否显示已完成任务的兼容字段，以及来源面板折叠状态。旧的单文件范围数据
会迁移为一张地图。来源文件或文件夹重命名时，插件同步迁移配置路径和无 ID 节点的位置键。

删除地图只删除配置，不删除笔记或任务。

## 5. 信息架构

主视图由四部分组成：

1. **顶部工具栏**：地图切换、来源管理、新建任务、自动规整、搜索、状态筛选、缩放和刷新。
2. **左侧来源面板**：Vault 文件树、来源勾选、当前文件和文件内任务。
3. **中央画布**：任务卡、内容卡、SVG 关系线、平移缩放、拖动和连线。
4. **右侧详情面板**：任务关系操作，或内容文件的快速编辑、预览和关联管理。

底部状态栏显示可见/总任务数、完成、进行中、可开始、被阻塞、异常和缺失来源数量，并提供关系任务
ID 的批量修复入口。

## 6. 核心交互

### 地图与来源

- 创建、重命名、切换和删除地图。
- 从 Vault 文件树选择多个 Markdown 文件或递归文件夹。
- 搜索来源树并在地图内选择当前文件。
- 来源创建、修改、删除和重命名后自动刷新。

### 画布

- 滚轮缩放，拖动空白区域平移。
- 拖动节点自由摆放并持久化坐标。
- “适应画布”计算当前节点边界。
- “规整”重建从左到右的分层布局并覆盖手动坐标。
- 单击选择，双击或按 Enter 打开原文。
- 空白处右键创建任务；节点右键提供关系、编辑、星标、打开与删除操作。

### 任务与关系

- 使用 Tasks 表单创建和编辑任务。
- 新任务自动获得兼容 Tasks Map 的六位小写字母数字 ID。
- 创建前置/后续任务时，必要时同时为现有任务补充 ID。
- 从右侧连接点拖到节点建立“当前节点 → 目标节点”关系。
- 从左侧连接点拖到节点建立“目标节点 → 当前节点”关系。
- 连接点拖到空白区域时，在放置位置创建对应的前置或后续任务。
- 点击依赖线，在确认后解除关系。
- 建立关系前阻止自连接、重复关系和新循环。
- 删除任务时，同时清理当前地图中后续任务对该 ID 的引用。
- 从任务底部连接点创建内容文件，或连接已有内容卡。
- 内容卡可以收起而不删除关联；删除内容文件前会批量清理全部任务关联。
- 快速编辑采用延迟保存和外部修改检测，冲突时禁止静默覆盖。

### 状态与元数据

- 待办点击后进入进行中，并在缺失时写入开始日期。
- 进行中点击后通过 Tasks API 标记完成。
- 已完成或已取消任务可以恢复。
- 详情面板支持添加/移除标签、切换星标和生成任务 ID。

## 7. 解析与图计算

`TaskParser` 按行识别 Markdown checkbox task，并维护当前标题路径。标准状态映射为待办、进行中、
完成和取消；其他单字符 checkbox 标记作为自定义状态保留。解析结果包含正文、原始正文、文件与行号、
状态、日期、标签、优先级、星标、ID、依赖 ID 和内容 WikiLink。

内容 WikiLink 会通过 Obsidian metadata cache 解析为真实 Vault 路径。解除、重命名和重绑时使用
“真实路径 + 原始别名”定位，兼容短 WikiLink 和相对链接。

`GraphBuilder` 首先统计任务 ID：

- 只有在当前地图中唯一的任务 ID 才可作为依赖目标；
- 重复 ID 的所有节点都会标记异常；
- 引用不存在或重复的 ID 会产生缺失引用；
- 深度优先遍历用于检测循环。

就绪状态：

```text
done 或 canceled                       -> finished
in_progress                            -> active
todo 且存在未完成/缺失的前置任务          -> blocked
其他 todo                              -> ready
```

完成或取消的前置任务都视为已解决。

## 8. 布局

内置布局使用固定节点尺寸和从左到右的分层算法：

- 每个节点层级等于其最深前置链层级加一；
- 同层优先继承父分支顺序，再按创建日期、路径和行号排序；
- 无依赖的独立任务进入下方三列网格；
- 循环节点集中到最后一个警告层；
- 用户拖动位置覆盖自动坐标，直到再次点击“规整”。

当前不依赖 Dagre 或 React。节点使用原生 HTML，边使用 SVG 贝塞尔曲线。

## 9. 写回安全

写回时优先按唯一任务 ID 重新定位，无法使用 ID 时再检查旧行号和原始正文。这样可减少文件发生变化后
写错行的风险。

跨文件关系及内容关联操作遵循：

1. 读取所有目标文件；
2. 在内存中顺序应用变换；
3. 写入每个文件；
4. 若后续写入失败，尝试恢复此前已写入文件的原文。

任务行写回只替换 checkbox 后的内容，保留原缩进、引用层级和列表标记。内容文件创建、移动、删除和
重绑还会同步恢复地图配置；无法安全覆盖快速编辑冲突时，操作会停止。

这不是版本控制或完整事务系统，因此用户仍应保持 Vault 备份。

## 10. 技术结构

```text
Vault Markdown
      │
      ▼
MapSourceResolver ──> 当前地图文件集合
      │
      ▼
TaskParser ──> TaskNode[]
      │
      ▼
GraphBuilder ──> 节点、依赖边、就绪状态与异常
      │
      ├──> LayeredLayout ──> 自动坐标
      │
      ▼
TaskGraphView ──> GraphCanvas + 来源面板 + 详情面板
      │
      ▼
TaskWriter / Tasks API ──> Vault Markdown
```

## 11. 当前验收基线

- 地图只包含已选来源中的 Markdown tasks。
- `🆔` / `⛔` 关系、箭头方向和就绪状态与原文一致。
- 文件变更后视图在 debounce 后刷新。
- 过期的异步刷新结果不能覆盖较新的地图状态；读取失败必须提示。
- 节点位置和视口按地图保存，来源重命名后配置仍可用。
- 新建、编辑、连接、解除、删除、标签、星标和状态操作只修改目标任务及必要引用。
- 重复 ID、缺失 ID、缺失引用和循环关系清晰可见。
- TypeScript 类型检查和生产打包通过。
- 核心 smoke test 覆盖解析、构图、布局、迁移、来源规则、关系元数据与失败回滚。
- 核心 smoke test 还覆盖短 WikiLink、嵌套任务结构、自定义状态和配置清理。
